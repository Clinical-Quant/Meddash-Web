# Decision Log: Meddash Ver 2.0 (Building UI Frontend & PostgreSQL Migration)

| Date | Decision | Rationale |
| :--- | :--- | :--- |
| 2026-03-21 | **Migrate Data Storage from SQLite to PostgreSQL** | Phase 2 scales operations significantly, involving real-time UI interactivity alongside multiple AI agents (e.g., Clinical Quant, crawler cron jobs, Twitter management bots). SQLite's file-level locking limits concurrent read/write and risks `database is locked` bottlenecks. PostgreSQL enables simultaneous transactions via row-level locking, ensuring multi-node architecture robustness. |
| 2026-03-21 | **Modify "Locked" Version 2.0 Architectural Diagram** | While strict directives normally prevent modifying the completed Version 1.0/2.0 diagrams, the strategic pivot to PostgreSQL is a foundational schema alteration. Overriding the lock is necessary to maintain an accurate ground-truth map of the evolving SaaS backend. |
| 2026-03-21 | **Implement FastAPI as Next.js to Python Bridge** | Continuing with the plan to decouple the Python analytic scripts from the React frontend, allowing FastAPI to govern PostgreSQL connections and trigger Python subprocess operations securely. |
| 2026-03-25 | **Adopt Campaign Sandbox (Pull ID) Isolation Workflow** | The current pipeline streams newly searched KOLs directly into the core `meddash_kols_db`, risking contamination with duplicates and removing the ability to isolate specific client requests. We will adopt a "Sandbox" model: New UI searches must include a `pull_id`. Results stream into a holding table (`kols_staging`). The UI groups by `pull_id`, allowing bulk disambiguation triggers and text report generation. Only verified KOLs push to the core `kols` table before the sandbox is wiped. |
| 2026-03-25 | **CSV-Array Pull IDs Instead of Row Cloning** | KOLs frequently overlap across disease areas and client campaigns. Cloning a full KOL row per `pull_id` would bloat `kols_staging` geometrically. Instead, `pull_id` is stored as a comma-separated text string (e.g. `"001,002,005"`). PostgreSQL `string_to_array()` and `array_remove()` functions parse these natively. Commit and Erase operations surgically add/remove individual IDs from the CSV without affecting other active Sandbox sessions sharing the same KOL. This architecture carries forward into the permanent `kols` table for historical campaign tracking. |
| 2026-03-26 | **Implement 3-Tier KOL Disambiguation Workflow** | A single-pass probabilistic scorer is insufficient for production-grade deduplication. Approved a multi-tier escalation model: **Tier 1 (Automatic)** — ORCID matches, high-confidence scored pairs (≥0.70), and standalone unique KOLs are auto-verified by the engine. **Tier 2 (HITL + Gemini Flash 2.5)** — Uncertain pairs are presented in a side-by-side UI comparison panel with AI-generated advisory reasoning from Gemini Flash. Human operator resolves as Same Person / Two People. **Tier 3 (Deep Disambiguation)** — Truly ambiguous pairs escalate to a `deep_disambiguation_needed` PostgreSQL table for manual research review. Expected Tier 3 volume: < 1% of total KOLs. This prevents database bloat from unresolved duplicate clones. |
